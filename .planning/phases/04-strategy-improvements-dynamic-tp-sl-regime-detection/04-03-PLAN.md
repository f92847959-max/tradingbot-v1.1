### Changes to scoring logic:

When `regime` is provided, adjust scoring weights per regime:

**Trend strength scoring (criterion 3, lines 67-77):**
- In TRENDING regime: boost trend scores (ADX is confirming our regime)
  - ADX >= 40: 15 (unchanged)
  - ADX >= 25: 12 (was 10)
  - ADX >= 15: 7 (was 5)
- In RANGING regime: reduce trend score importance (trend is not the play)
  - ADX >= 40: 10 (was 15)
  - ADX >= 25: 7 (was 10)
  - ADX >= 15: 3 (was 5)
- In VOLATILE regime: keep original weights (neutral)

**Volatility scoring (criterion 5, lines 84-95):**
- In VOLATILE regime: narrow the "ideal" ATR ratio band and cap score
  - 0.8 <= ratio <= 1.2: 10 (was 15 -- penalize slightly, conditions are riskier)
  - 0.5 <= ratio <= 2.0: 5 (was 8)
- In TRENDING/RANGING: keep original weights

This is a light touch -- the regime-aware parameter table (min_score, min_confidence)
does the heavy lifting. Scoring weight adjustments are secondary refinement.

## File 4: `risk/position_sizing.py` (MODIFY)

### Add ATR guard method:

```python
def calculate_with_atr_guard(
    self,
    equity: float,
    entry_price: float,
    stop_loss: float,
    atr: float,
    max_atr_for_trading: float = 5.0,
) -> float:
    """Position sizing with ATR guard for extreme volatility.

    When ATR exceeds max_atr_for_trading, returns minimum lot size
    instead of calculated size. This prevents large positions during
    extreme market events.

    Args:
        equity: Account equity
        entry_price: Planned entry price
        stop_loss: Planned stop loss price
        atr: Current ATR-14 value
        max_atr_for_trading: Maximum ATR threshold (default 5.0 for Gold)

    Returns:
        Lot size (minimum if ATR exceeds guard)
    """
    if atr > max_atr_for_trading:
        logger.warning(
            "ATR %.2f exceeds max %.2f -- using minimum lot size",
            atr, max_atr_for_trading,
        )
        return self.min_lot_size

    return self.calculate(equity, entry_price, stop_loss)
```

The existing `calculate()` method is unchanged. When SL is ATR-based (from
`calculate_sl_tp` with ATR multipliers), position size ALREADY scales inversely
with ATR because `sl_distance = atr * multiplier`. The ATR guard is an additional
safety net for extreme conditions.

<notes>
- The StrategyManager evaluate() restructuring MUST keep backward compatibility: when
  `mtf_data` is None or missing "5m" key, regime defaults to RANGING (safest) and
  existing behavior is preserved.

- TradeScorer.score() MUST still work when `regime=None` (backward compat for tests
  and any code path that doesn't pass regime). When None, use original scoring weights.

- The `calculate_sl_tp_for_regime` function uses a lazy import of `get_regime_params`
  to avoid circular imports (entry_calculator.py is in strategy/, regime_params.py is
  also in strategy/ -- not circular, but the lazy import pattern keeps the module
  independent).

- The ATR guard threshold of 5.0 for Gold means: if ATR-14 > $5.00, use minimum
  position size. Normal Gold ATR is ~$1-3, so 5.0 triggers only during extreme events
  (NFP, FOMC, black swan). This is a safety net, not a regular occurrence.

- Do NOT modify `ensemble.py` or `trading_loop.py` in this plan. The regime-aware
  SL/TP calculation is available via `calculate_sl_tp_for_regime()`, but wiring it
  into the ensemble predictor's live prediction flow is a separate concern that
  should be done carefully to avoid breaking live trading. The ensemble predictor
  already has its own ATR-based SL/TP calculation. For now, the regime-aware
  params are used in StrategyManager (signal approval/rejection) and available
  for future use in the ensemble.

- The StrategyManager changes affect the signal approval pipeline. Signals that
  were previously approved may now be rejected (e.g., 0.70 confidence in VOLATILE
  regime with min_confidence=0.80). This is intentional -- the system should be
  more selective in volatile conditions.
</notes>
