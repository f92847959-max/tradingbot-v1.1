# Phase 12.4: Exit AI Training & Baseline Evaluation - Research

**Researched:** 2026-04-24
**Domain:** Smart-exit specialist training / trade-lifecycle supervision / walk-forward baseline comparison
**Confidence:** HIGH

---

## Summary

Phase 10 already built a deterministic Smart Exit Engine with four distinct decision
surfaces: dynamic stop loss (`dynamic_sl.py`), dynamic take profit (`dynamic_tp.py`),
ATR trailing (`trailing_manager.py`), and TP1 partial close (`partial_close.py`).
That makes Exit AI a strong follow-up candidate because the repo already has a rule-based
baseline that can act as teacher, fallback, and evaluation anchor.

The correct 12.4 scope is not "replace exits with black-box ML". The correct scope is:

1. build causal exit snapshots from open trades plus candle context,
2. map them into a constrained action space,
3. train a separate Exit-AI specialist model,
4. compare it leakage-safely against the existing Smart Exit Engine baseline.

Current runtime architecture is entry-centric: `trading/signal_generator.py` opens trades,
while `order_management/order_manager.py` and `position_monitor.py` manage lifecycle
changes after entry. That means 12.4 should stay offline and evaluation-focused. Runtime
application belongs in the next phase (12.5).

**Primary recommendation:** Create a separate Exit-AI specialist under
`ai_engine/saved_models/specialists/exit_ai/`, trained on trade snapshots with the action
set `HOLD`, `TIGHTEN_SL`, `PARTIAL_CLOSE`, `FULL_EXIT`, and validate it via walk-forward
comparison against the deterministic baseline before any live integration.

## Recommended Action Space

The Exit-AI should be trained on a constrained action space:

- `HOLD` - keep current exit configuration unchanged
- `TIGHTEN_SL` - move stop closer in the favorable direction only
- `PARTIAL_CLOSE` - close a fraction using the existing TP1/partial-close semantics
- `FULL_EXIT` - flatten the position early

Do not include `WIDEN_SL`, `ADD_TO_POSITION`, or any entry action. Those violate the
existing risk architecture.

### Model Choice

Use a separate gradient-boosted classifier first, preferably LightGBM or XGBoost,
because the repo already uses both and the training/evaluation tooling is built around
them. Do not introduce reinforcement learning, sequence models, or new dependencies in
12.4.

## Architecture Patterns

### Pattern 1: Specialist storage, not core overwrite

Use the same separation principle as Phase 12.3:

```text
ai_engine/saved_models/
  specialists/
    exit_ai/
      production.json
      v001_.../
        exit_ai_model.pkl
        exit_ai_scaler.pkl
        exit_ai_version.json
        exit_action_manifest.json
```

### Pattern 2: Offline supervisor derived from live lifecycle rules

The label builder should consume:

- trade entry / direction / current stop / current TP
- recent candles and indicators
- outputs from deterministic exit modules
- realized or counterfactual trade outcomes for evaluation

This keeps the Exit-AI anchored to the existing Smart Exit Engine rather than inventing
an unrelated objective.

### Pattern 3: Baseline-vs-specialist comparison

Every walk-forward window should preserve:

- baseline rule-only path
- baseline + Exit-AI candidate path
- identical window boundaries and scaling discipline

The accepted evaluation is comparative, not absolute.

## Common Pitfalls

### Pitfall 1: Label leakage from future trade outcome

If an exit label is assigned using future max excursion without a delayed/causal
snapshot definition, the model will look unrealistically strong.

**Avoidance:** snapshot each open trade at a decision timestamp and generate labels from
information available at that timestamp plus a clearly defined evaluation horizon.

### Pitfall 2: HOLD dominates every class distribution

Trade-management data will heavily skew toward "do nothing".

**Avoidance:** track class ratios, use stratified reporting, and reject any model that
wins by predicting HOLD almost everywhere.

### Pitfall 3: Exit AI learns to widen risk indirectly

If labels or metrics reward longer survival without risk constraints, the learned policy
may prefer looser exits.

**Avoidance:** action space must exclude risk-widening and evaluation must penalize
drawdown expansion.

### Pitfall 4: Training schema mismatches runtime schema

If the model is trained on features unavailable during live trade management, runtime
integration will silently degrade or fail.

**Avoidance:** keep the feature manifest explicit and runtime-loadable from the artifact.
