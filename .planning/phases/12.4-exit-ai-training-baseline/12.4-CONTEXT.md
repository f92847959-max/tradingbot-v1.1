# Phase 12.4: Exit AI Training & Baseline Evaluation - Context

**Gathered:** 2026-04-24
**Status:** Ready for planning
**Mode:** Smart-discuss defaults applied for autonomous planning

<domain>
## Phase Boundary

Phase 12.4 adds a separate Exit-AI specialist that learns trade-management actions from
the existing Smart Exit Engine and trade lifecycle data.

The scope is:
- causal exit snapshots from open trades plus candle context
- constrained action labels: `HOLD`, `TIGHTEN_SL`, `PARTIAL_CLOSE`, `FULL_EXIT`
- separate specialist training, versioning, and feature/action manifest
- walk-forward comparison against the deterministic Smart Exit Engine baseline
- promotion/calibration gate for a later runtime rollout

Out of scope for 12.4:
- live broker-side execution changes
- widening risk or creating new entries
- replacing the deterministic Smart Exit Engine

</domain>

<decisions>
## Implementation Decisions

### Exit AI Role
- Exit AI is an advisory specialist for trade management, not a replacement for entry generation.
- The deterministic Smart Exit Engine from Phase 10 remains the baseline, teacher, and hard fallback.
- 12.4 stays offline/training-only; runtime application is deferred to 12.5.

### Action Space
- Allowed actions are `HOLD`, `TIGHTEN_SL`, `PARTIAL_CLOSE`, and `FULL_EXIT`.
- No action may widen stop loss, add risk, pyramid, or open new trades.
- `PARTIAL_CLOSE` must reuse the existing TP1/partial-close semantics instead of inventing a new broker action model.

### Leakage Discipline
- Every training sample must come from a clearly defined decision timestamp while the trade is still open.
- Labels must be generated from data that would have been available at that timestamp plus a documented evaluation horizon.
- Future-row or post-close information may be used only in the supervised target construction, never as direct model input.

### Artifact and Storage Layout
- Exit AI artifacts live under `ai_engine/saved_models/specialists/exit_ai`.
- Artifacts include model file, scaler, version metadata, and action/feature manifests.
- Core ensemble artifacts and specialist 12.3 artifacts must not be overwritten.

### Evaluation Gate
- Promotion requires baseline comparison over identical walk-forward windows.
- Success is not raw accuracy; it is a documented trade-off between drawdown protection, upside retention, and early-exit cost.
- HOLD-dominant degenerate models are not acceptable even if they show apparent drawdown reduction.

### the agent's Discretion
- Exact feature names for the exit snapshot vector
- Whether the specialist uses LightGBM, XGBoost, or a small stacked comparison
- Exact metric thresholds for promotion, as long as they remain documented and baseline-relative

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `exit_engine/dynamic_sl.py` exposes the tighten/structure semantics
- `exit_engine/dynamic_tp.py` exposes TP/TP1 and structure context
- `exit_engine/trailing_manager.py` encodes monotonic tighten-only trailing behavior
- `exit_engine/partial_close.py` already defines the one-shot partial-close contract
- `order_management/order_manager.py` owns broker-side modify/close orchestration
- `order_management/position_monitor.py` owns lifecycle state transitions
- `ai_engine/training/walk_forward.py` provides leakage-safe time-series validation
- `ai_engine/training/model_versioning.py` provides versioned artifact storage patterns

### Established Patterns
- specialist artifacts are isolated from core artifacts
- walk-forward windows with train-only fitting are mandatory for performance claims
- focused pytest files per subsystem are the norm
- optional runtime components default to no-op when artifacts are missing

</code_context>

<specifics>
## Specific Ideas

- Build a dedicated exit snapshot builder instead of embedding label logic in `order_manager.py`.
- Keep an explicit `exit_action_manifest.json` so runtime knows which actions the model was trained to emit.
- Compare rule-only and rule-plus-specialist behavior on the same trade windows.
- Reuse 12.1 calibration style for confidence and promotion metadata instead of inventing a new governance format.

</specifics>

<deferred>
## Deferred Ideas

- Reinforcement learning or direct policy optimization
- Broker-level execution policy search
- Cross-asset or portfolio-level exit optimization
- Live auto-application of Exit-AI outputs before 12.5 governance exists

</deferred>
