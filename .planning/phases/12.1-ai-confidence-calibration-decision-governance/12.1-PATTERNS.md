## Reuse Patterns

### 1. Versioned JSON artifacts

Use `ai_engine/training/model_versioning.py` as the source pattern.

- `create_version_dir()` already defines the version folder lifecycle.
- `write_version_json()` is the correct shape for persisted metadata.
- `update_production_pointer()` is the pattern for promoting a trained artifact to production on Windows without symlinks.

Implication for Phase 12.1:
- Store `calibration.json` and `thresholds.json` inside the version directory beside existing model files.
- Extend `version.json` metadata instead of inventing a second registry layer.

### 2. Leakage-safe training flow

Use `WalkForwardValidator.run_window()` in `ai_engine/training/walk_forward.py` as the core pattern.

- The repo already does `train -> val -> test` splitting inside each window.
- Fresh scalers are fit per window on training data only.
- Validation is already the correct place for threshold tuning before test evaluation.

Implication for Phase 12.1:
- Calibrator fitting must happen after base model training and before final test scoring.
- Threshold tuning must consume validation/calibration outputs, not the final test slice.

### 3. Existing gate logic extraction

Use `EnsemblePredictor` in `ai_engine/prediction/ensemble.py` as the migration source.

Key existing fields and methods:
- `min_confidence`
- `decision_threshold`
- `max_conflict_ratio`
- `_compute_conflict_ratio(...)`
- current pass/block behavior around final action and confidence

Implication for Phase 12.1:
- Do not duplicate ensemble aggregation.
- Extract the final governance policy into a pure `DecisionGovernor` that accepts already-computed scores, regime, conflict ratio, and optional specialist evidence.
- Keep `EnsemblePredictor` responsible for aggregation, but delegate the final gate decision.

### 4. Typed regime/value objects

Use `strategy/regime_detector.py` as the pattern for lightweight typed domain objects.

- `MarketRegime` enum is the canonical regime vocabulary.
- `RegimeState` shows the repo's small dataclass style for structured runtime state.

Implication for Phase 12.1:
- Governance types should use dataclasses/enums, not loose dicts everywhere.
- Threshold lookup should consume existing regime labels instead of defining a second regime taxonomy.

### 5. JSON ORM persistence

Use `Signal` and `RiskEvent` patterns in `database/models.py`.

- `JSON_COMPAT` is already the cross-database JSON field pattern.
- Audit-like payloads belong in structured JSON columns plus indexed timestamps/status fields.

Implication for Phase 12.1:
- Governance audit logging should be a dedicated ORM model or a careful extension of existing signal persistence.
- Persist HOLD/rejected outcomes, gate reasons, artifact version, threshold source, and conflict ratio.

## Preferred Plan Boundaries

Recommended split for planning:

1. Training/calibration artifact layer
   Files: `ai_engine/calibration/*`, `ai_engine/training/walk_forward.py`, `ai_engine/training/pipeline.py`, `ai_engine/training/model_versioning.py`, tests for calibration and threshold tuning

2. Runtime governance layer
   Files: `ai_engine/governance/*`, `ai_engine/prediction/ensemble.py`, `strategy/regime_detector.py`, tests for decision governor and specialist veto rules

3. Persistence/monitoring layer
   Files: `database/models.py`, repository additions, `trading/trading_loop.py` and/or signal persistence path, tests for decision logging and champion/challenger metadata

## Avoid

- Do not add calibration math directly inside `EnsemblePredictor`.
- Do not fit calibrators on test/OOS slices.
- Do not create a second regime enum or second model registry abstraction.
- Do not let specialist evidence create trades without a core non-HOLD action.
