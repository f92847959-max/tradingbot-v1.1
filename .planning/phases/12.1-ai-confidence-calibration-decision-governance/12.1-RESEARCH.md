### Pattern 1: Calibrate After Training, Before Thresholding

**What:** Fit a post-hoc calibrator on data that was not used to fit the base classifier, then use calibrated probabilities for thresholds and governance. [CITED: https://scikit-learn.org/stable/modules/calibration.html]

**When to use:** Every walk-forward window, after XGBoost/LightGBM fit and before threshold selection. [VERIFIED: codebase `ai_engine/training/walk_forward.py`]

**Example:**

```python
from sklearn.calibration import CalibratedClassifierCV
from sklearn.frozen import FrozenEstimator


def fit_prefit_calibrator(base_model, X_calib, y_calib_mapped):
    """Source: scikit-learn CalibratedClassifierCV + FrozenEstimator docs."""
    calibrator = CalibratedClassifierCV(
        estimator=FrozenEstimator(base_model),
        method="temperature",
    )
    calibrator.fit(X_calib, y_calib_mapped)
    return calibrator
```

### Pattern 2: Separate Statistical Probability From Trading Decision

**What:** Treat calibrated probability as model belief and thresholds as a downstream trading utility policy. [CITED: https://scikit-learn.org/stable/modules/classification_threshold.html]

**When to use:** Thresholds are chosen from validation trading metrics, not from hard-coded 0.55/0.15 values. [VERIFIED: codebase `ai_engine/prediction/ensemble.py`; VERIFIED: codebase `ai_engine/training/trade_filter.py`]

**Example:**

```python
def choose_threshold(candidate_metrics):
    """Source: existing trade_metrics_rank pattern in ai_engine/training/trade_filter.py."""
    return max(
        candidate_metrics,
        key=lambda item: (
            item["n_trades"] >= item["min_trades_target"],
            item["profit_factor"],
            item["win_rate"],
            item["expectancy"],
            item["total_pips"],
        ),
    )
```

### Pattern 3: Pure Decision Governor

**What:** A deterministic pure function consumes already computed evidence and returns an auditable action. [VERIFIED: codebase `ai_engine/prediction/ensemble.py`; VERIFIED: codebase `strategy/regime_detector.py`]

**When to use:** Live inference and test fixtures; do not fit, mutate, fetch, or write inside the governor. [VERIFIED: codebase `CLAUDE.md`]

**Example:**

```python
from dataclasses import dataclass, field


@dataclass(frozen=True)
class DecisionAudit:
    raw_action: str
    final_action: str
    regime: str
    calibrated_confidence: float
    threshold_used: float
    conflict_ratio: float
    gate_reasons: list[str] = field(default_factory=list)
    specialist: dict | None = None


def govern(core_action, calibrated_confidence, threshold, conflict_ratio, regime):
    reasons: list[str] = []
    final_action = core_action
    if core_action != "HOLD" and calibrated_confidence < threshold:
        final_action = "HOLD"
        reasons.append("calibrated_confidence_below_threshold")
    return DecisionAudit(
        raw_action=core_action,
        final_action=final_action,
        regime=regime,
        calibrated_confidence=calibrated_confidence,
        threshold_used=threshold,
        conflict_ratio=conflict_ratio,
        gate_reasons=reasons,
    )
```

### Pattern 4: Specialist Evidence Cannot Originate Trades

**What:** Specialist outputs modify a non-HOLD core decision only; if the core ensemble is HOLD, final action remains HOLD. [VERIFIED: codebase `.planning/ROADMAP.md`]

**When to use:** Implement the interface in Phase 12.1 and wire real specialist outputs in Phase 12.3. [VERIFIED: codebase `.planning/ROADMAP.md`]

def apply_specialist_policy(core_action: str, specialist: dict | None) -> tuple[str, list[str]]:
    if core_action == "HOLD" or not specialist:
        return core_action, ["specialist_no_standalone_trade"]
    direction = specialist.get("direction", "HOLD")
    if direction == "HOLD":
        return core_action, ["specialist_neutral"]
    if direction == core_action:
        return core_action, ["specialist_confirms"]
    return "HOLD", ["specialist_contradiction_veto"]
```

- **Calibrating on test data:** Test windows must remain for reporting only; scikit-learn warns that fitting on non-independent data biases calibration. [CITED: https://scikit-learn.org/stable/modules/calibration.html; CITED: https://scikit-learn.org/stable/common_pitfalls.html]
- **Replacing regime logic with a second implementation:** Existing regime classification already has hysteresis and a RANGING fallback. [VERIFIED: codebase `strategy/regime_detector.py`; VERIFIED: codebase `.planning/STATE.md`]
- **Letting a specialist model create trades alone:** Phase 12.3 explicitly says the specialist may confirm/weaken/veto but must not trade alone. [VERIFIED: codebase `.planning/ROADMAP.md`]
- **Persisting only final BUY/SELL decisions:** The current trading loop returns early for HOLD decisions, so gate rejections would disappear unless a decision-log path is added. [VERIFIED: codebase `trading/trading_loop.py`]

## Common Pitfalls

### Pitfall 1: Calibrating on the Same Data Used for Model Fit

**What goes wrong:** Probability estimates become overconfident and live hit rates miss reported calibration. [CITED: https://scikit-learn.org/stable/modules/calibration.html]

**Why it happens:** The calibrator sees model outputs from data the model already trained on. [CITED: https://scikit-learn.org/stable/modules/calibration.html]

**How to avoid:** Fit the base model on `X_train`, fit calibrator on `X_val` or OOS calibration folds, and reserve `X_test` for reporting. [VERIFIED: codebase `ai_engine/training/walk_forward.py`; CITED: https://scikit-learn.org/stable/common_pitfalls.html]

**Warning signs:** Brier/log-loss improves on calibration data but not on test windows. [CITED: https://scikit-learn.org/stable/modules/calibration.html]

### Pitfall 2: Per-Regime Calibrators With Tiny Samples

**What goes wrong:** RANGING/TRENDING/VOLATILE calibrators overfit and create unstable live thresholds. [ASSUMED]

**Why it happens:** Existing minimum train size is 1500 and validation is 15%, so per-regime validation samples can be small. [VERIFIED: codebase `.planning/STATE.md`; VERIFIED: codebase `ai_engine/training/walk_forward.py`]

**How to avoid:** Use one global per-model calibrator first; tune regime-specific thresholds on pooled walk-forward outputs. [CITED: https://scikit-learn.org/stable/modules/generated/sklearn.calibration.CalibratedClassifierCV.html; VERIFIED: codebase `ai_engine/training/trade_filter.py`]

**Warning signs:** A regime threshold is selected from fewer than the required `min_trades_target`. [VERIFIED: codebase `ai_engine/training/trade_filter.py`]

### Pitfall 3: Confusing Confidence With Trading Utility

**What goes wrong:** A calibrated 0.65 probability is treated as automatically tradable even if expected value after spread/slippage is negative. [CITED: https://scikit-learn.org/stable/modules/classification_threshold.html; VERIFIED: codebase `ai_engine/training/evaluation.py`]

**Why it happens:** Probability estimation and action thresholding are separate problems. [CITED: https://scikit-learn.org/stable/modules/classification_threshold.html]

**How to avoid:** Tune thresholds by trading metrics, using the existing ranking pattern: enough trades, profit factor, win rate, expectancy, total pips. [VERIFIED: codebase `ai_engine/training/trade_filter.py`]

**Warning signs:** High accuracy/F1 but low profit factor or very low trade count. [VERIFIED: codebase `ai_engine/training/walk_forward.py`]

### Pitfall 4: Ignoring Probability Distortion From Weights

**What goes wrong:** Class weighting or sample weighting improves classification balance but makes raw probabilities less reliable. [CITED: https://lightgbm.readthedocs.io/en/latest/pythonapi/lightgbm.LGBMClassifier.html]

**Why it happens:** LightGBM docs explicitly warn that weights can produce poor individual class probability estimates. [CITED: https://lightgbm.readthedocs.io/en/latest/pythonapi/lightgbm.LGBMClassifier.html]

**How to avoid:** Keep weighting if it improves trading outcomes, but calibrate probabilities after weighting. [CITED: https://lightgbm.readthedocs.io/en/latest/pythonapi/lightgbm.LGBMClassifier.html; CITED: https://scikit-learn.org/stable/modules/calibration.html]

**Warning signs:** Raw `predict_proba` confidence is high while reliability bins show lower realized win rates. [CITED: https://scikit-learn.org/stable/modules/calibration.html]

### Pitfall 5: Label Order Mismatch

**What goes wrong:** Calibration or thresholds interpret class 0 as HOLD instead of SELL. [VERIFIED: codebase `ai_engine/models/xgboost_model.py`; VERIFIED: codebase `ai_engine/models/lightgbm_model.py`]

**Why it happens:** Training labels use signal space `[-1, 0, 1]`, model class space `[0, 1, 2]`, and ensemble action labels `SELL/HOLD/BUY`. [VERIFIED: codebase `ai_engine/training/trade_filter.py`; VERIFIED: codebase `ai_engine/prediction/ensemble.py`]

**How to avoid:** Persist `class_labels = ["SELL", "HOLD", "BUY"]` and mapped integer order in `calibration.json`. [VERIFIED: codebase `ai_engine/prediction/ensemble.py`]

**Warning signs:** BUY/Sell probabilities appear swapped in calibration reports. [ASSUMED]

### Pitfall 6: Losing HOLD and Rejection Decisions

**What goes wrong:** Governance cannot be audited because only executed/rejected trade attempts are saved. [VERIFIED: codebase `trading/trading_loop.py`; VERIFIED: codebase `trading/signal_generator.py`]

**Why it happens:** The trading loop returns immediately for HOLD raw signals. [VERIFIED: codebase `trading/trading_loop.py`]

**How to avoid:** Persist a compact AI decision log for every governed signal before the strategy/risk layers return. [VERIFIED: codebase `database/models.py`; VERIFIED: codebase `CLAUDE.md`]

**Warning signs:** Daily signal counts are much lower than trading-loop ticks. [ASSUMED]

### Pitfall 7: Specialist Veto Becomes Specialist Trading

**What goes wrong:** A future specialist can create BUY/SELL signals when the core ensemble would HOLD. [VERIFIED: codebase `.planning/ROADMAP.md`]

**Why it happens:** A naive weighted ensemble treats every model as a peer. [ASSUMED]

**How to avoid:** Encode an invariant: `core_action == HOLD -> final_action == HOLD`, regardless of specialist direction. [VERIFIED: codebase `.planning/ROADMAP.md`]

**Warning signs:** Tests show a specialist-only BUY/SELL path. [VERIFIED: codebase `.planning/ROADMAP.md`]

### Calibration Metrics Table

from sklearn.metrics import brier_score_loss, log_loss


CLASS_LABELS = [0, 1, 2]  # SELL, HOLD, BUY


def calibration_metrics(y_true_mapped, calibrated_probs):
    """Source: scikit-learn brier_score_loss/log_loss documentation."""
    return {
        "brier_score": float(
            brier_score_loss(
                y_true_mapped,
                calibrated_probs,
                labels=CLASS_LABELS,
                scale_by_half=False,
            )
        ),
        "log_loss": float(log_loss(y_true_mapped, calibrated_probs, labels=CLASS_LABELS)),
    }
```

## State of the Art

| Old Approach | Current Approach | When Changed / Source | Impact |
|--------------|------------------|------------------------|--------|
| Fixed `min_confidence` and raw max probability | Calibrated probabilities plus tuned decision thresholds | scikit-learn docs separate probability estimation from decision thresholding. [CITED: https://scikit-learn.org/stable/modules/classification_threshold.html] | Planner should create both calibration and threshold-selection tasks. |
| Binary-only threshold examples | Custom multiclass trading threshold table | `TunedThresholdClassifierCV` is documented around thresholding classifier scores, while this project has BUY/HOLD/SELL and trading utility. [CITED: https://scikit-learn.org/stable/modules/classification_threshold.html; VERIFIED: codebase `ai_engine/training/trade_filter.py`] | Use project-specific tuner, not generic binary threshold wrapper. |
| Sigmoid/isotonic only | Temperature scaling available in scikit-learn 1.8 | `method="temperature"` added in scikit-learn 1.8. [CITED: https://scikit-learn.org/stable/modules/generated/sklearn.calibration.CalibratedClassifierCV.html] | Prefer temperature for multiclass default in this installed environment. |
| Model registry stages | Model aliases/tags for champion/challenger | MLflow docs describe aliases such as champion and tags such as validation status. [CITED: https://www.mlflow.org/docs/3.0.1/model-registry] | Project can mimic aliases locally with `production.json` and version metadata. |
| Averaging profit factors | Aggregate gross profit / gross loss | Existing project decision says aggregate PF from totals. [VERIFIED: codebase `.planning/STATE.md`] | Threshold selection should preserve that metric logic. |

**Deprecated/outdated:**

- Treating raw tree model probabilities as reliable confidence is outdated for this phase because calibration is explicitly required and LightGBM docs warn weights can degrade probability estimates.
