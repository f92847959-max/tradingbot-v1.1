# Phase 12.7 - Research

Research question: what needs to be known to plan AI Training Pipeline Hardening without changing live runtime behavior.

## Existing Training Architecture

- `scripts/train_models.py` is the operator entrypoint. It currently supports `--broker`, `--csv`, and `--synthetic`, then calls `ModelTrainer.train_all()` or `ModelTrainer.train_from_csv()`.
- `scripts/train_models.py` can fetch broker candles with `get_candles_paginated(total_count=...)` and prints `Received {n} candles` / `Saved {n} candles`, but it does not preserve a structured row-loss manifest.
- `ai_engine/training/data_source.py` already has the better source abstraction: `load_from_db()`, `load_from_file()`, `load_auto()`, `DataLoadResult`, and `summarize_dataframe()`.
- `market_data/historical.py` and `database/repositories/candle_repo.py` provide the DB-backed historical candle path. This is the correct backfill source for substantial training, because the broker API can return fewer candles than requested.
- `ai_engine/training/pipeline.py` runs validation, feature creation, label generation, warmup trimming, feature/label extraction, walk-forward validation, and model saving.
- `ai_engine/training/data_preparation.py` validates duration after data is already loaded, feature/label work is near, and the error reports coarse months. That is too late for the known broker-short-history failure mode.
- `ai_engine/training/walk_forward.py` already provides expanding-window validation, per-window calibration, threshold tuning, and training report generation.
- `ai_engine/training/model_versioning.py` already provides version directories, `production.json`, and isolated specialist roots.
- `ai_engine/training/specialist_pipeline.py` and `ai_engine/training/decision_head_pipeline.py` are useful examples for specialist artifact isolation and candidate-vs-baseline comparison discipline.

## Data Coverage and Source Strategy

Known failure mode: a broker-backed run requested more history but only received 1216 candles, saved 520 candles, and later failed because the trainable span was about two days. The plan must prevent this from happening late.

Required strategy:

- Prefer DB-backed historical candles for real backfills through `data_source.load_auto()` or `load_from_db()`.
- Treat broker fetches as fresh incremental data, not the default source for multi-month backfills.
- Add an early preflight before feature creation, label generation, SHAP, walk-forward, or artifact writing.
- Report actual span in days/hours in addition to months, using `start_timestamp`, `end_timestamp`, `rows`, `timeframe`, and `min_required_days`.
- Preserve row counts across source stages: `received`, `saved`, `normalized`, `feature_ready`, and `label_ready`.
- Write a dataset manifest next to training reports so failed and successful runs have an audit trail.

## Labeling and Leakage Boundaries

Existing `LabelGenerator` creates entry labels using triple-barrier logic with spread/slippage. Phase 12.7 should not replace it. It should wrap and document it in a causal label contract.

Needed labels:

- `entry_label`: existing `-1/0/1` SELL/HOLD/BUY target from the triple-barrier generator.
- `abstain_label`: explicit HOLD/ABSTAIN reason target, not a garbage class for missing features or blocked states.
- `exit_label`: optional exit/trade-management label sourced from existing exit AI artifacts when available, but not required for core training to run.
- `confidence_target`: bucketed outcome quality used for calibration and confidence-bucket promotion checks.
- `risk_bucket`: coarse risk context derived from ATR/spread/volatility columns using only current or past rows.

Leakage rule: every label may inspect future outcome only as the target horizon, never as an input feature. Features and scalers must still be fit inside walk-forward windows.

## Walk-forward, Calibration, Promotion

`walk_forward.py` already creates the base report, calibrators, threshold artifacts, and final production models. Phase 12.7 should add a hard promotion decision on top of those outputs.

Promotion must compare candidate and champion on identical windows and net-of-cost trading metrics:

- candidate profit factor must be at least 1.20 and at least 5 percent better than champion, unless champion has no trades and candidate passes all other gates
- candidate max drawdown may not exceed champion drawdown by more than 10 percent
- candidate calibration error must be <= 0.08 for actionable confidence buckets
- each promoted confidence bucket must have minimum support of 20 samples
- candidate must keep enough non-HOLD decisions to avoid a fake "safe" model that only abstains
- failed gates must block `production.json` updates and still write `promotion_decision.json`

This phase should produce offline/shadow artifacts only. Runtime selection remains governed by later or existing runtime phases.

## Validation Architecture

Targeted commands:

- `python -m pytest tests/test_training_data_coverage.py -q`
- `python -m pytest tests/test_causal_training_labels.py -q`
- `python -m pytest tests/test_training_promotion_gate.py -q`
- `python -m pytest tests/test_training_data_coverage.py tests/test_causal_training_labels.py tests/test_training_promotion_gate.py tests/test_walk_forward_calibration.py -q`

Acceptance focus:

- Short real-span data aborts before fitting.
- Row-loss diagnostics distinguish received, saved, normalized, feature-ready, and label-ready rows.
- Label generation is future-mutation safe for feature columns.
- Candidate/champion comparison uses identical walk-forward windows.
- Failed promotion gates never update production pointers.

## Risks and Non-goals

- Non-goal: do not wire a new model into live trading runtime.
- Non-goal: do not replace the existing `LabelGenerator`; wrap it with a clearer manifest and extra label columns.
- Non-goal: do not make broker history the reliable backfill path.
- Risk: `scripts/train_models.py` CLI changes can break existing operator habits; preserve legacy flags and add new source controls compatibly.
- Risk: over-strict promotion gates can block every model; write decisions with reasons so the operator can tune thresholds later.
