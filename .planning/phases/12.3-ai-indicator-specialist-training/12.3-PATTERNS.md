## Reuse Patterns

### 1. Causal feature families

Use `ai_engine/features/support_resistance.py` as the main source pattern.

- It already exposes a dedicated class with `FEATURE_NAMES`.
- It computes structure-aware signals from candles.
- It uses delayed/shifted pivot semantics, which is critical for leakage-safe specialist features.

Implication for 12.3:
- `market_structure_liquidity` should be a dedicated feature class, not loose helper functions.
- Feature names must be explicit and versionable.

### 2. Optional side-branch ensemble integration

Use `ai_engine/prediction/ensemble.py` as the runtime integration pattern.

- Sentiment and MiroFish already behave as optional extra voices.
- `gate_reasons` and `final_aggregation` are already structured outputs.

Implication for 12.3:
- Specialist integration should be an optional overlay that preserves baseline behavior when disabled.
- Add specialist outputs to `final_aggregation` instead of rewriting the whole ensemble contract.

### 3. Versioned model artifacts

Use `ai_engine/training/model_versioning.py` as the persistence pattern.

- Version directories and `production.json` already exist.
- Windows-safe JSON pointer logic is already chosen.

Implication for 12.3:
- Create specialist-specific pointer/update helpers instead of reusing the core copier blindly.
- Never overwrite core `xgboost_gold.pkl`, `lightgbm_gold.pkl`, or core scaler files during specialist promotion.

### 4. Walk-forward and per-window scaling

Use `ai_engine/training/walk_forward.py` as the time-series validation pattern.

- It already enforces expanding windows and a purge gap.
- It already fits fresh scalers per window.

Implication for 12.3:
- Specialist training must follow the same discipline.
- Event-like specialist features must be computed in a window-safe or delayed fashion before any uplift claim is accepted.

## Preferred Plan Boundaries

1. Feature block and leakage tests
   Files: `ai_engine/features/market_structure_liquidity.py`, `feature_engineer.py`, `tests/test_market_structure_liquidity.py`

2. Specialist model training and storage
   Files: training/pipeline or specialist-specific training helpers, model versioning extensions, `tests/test_specialist_model.py`, `tests/test_specialist_walk_forward.py`

3. Runtime overlay and logging
   Files: `ai_engine/prediction/ensemble.py` or specialist adapter, `tests/test_ensemble_specialist.py`

## Avoid

- Do not fold specialist training into the core model artifact path.
- Do not reuse the core scaler for specialist features.
- Do not let specialist-disabled mode change core outputs.
- Do not enable specialist runtime influence without a governance-compatible gate path.
