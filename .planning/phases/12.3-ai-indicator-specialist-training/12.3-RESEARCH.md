## Summary

Use a Market-Structure/Liquidity-Sweep/Fair-Value-Gap specialist as the second AI candidate. This is the best fit because the project already has causal swing support/resistance features and a microstructure scaffold, while true order-book data is deferred to Phase 13 and Elliott Wave is deferred to Phase 14. [VERIFIED: ai_engine/features/support_resistance.py; VERIFIED: ai_engine/features/microstructure_features.py; VERIFIED: .planning/ROADMAP.md]

The specialist should be a separate LightGBM-backed classifier trained only on the specialist feature block, with its own scaler, version metadata, trade-filter thresholds, and production pointer. This keeps the current XGBoost/LightGBM core ensemble load path intact and makes the specialist a governed companion voice rather than a replacement model. [VERIFIED: ai_engine/models/lightgbm_model.py; VERIFIED: ai_engine/training/model_versioning.py; VERIFIED: ai_engine/prediction/ensemble.py]

The plan must treat leakage as the primary risk. The current pipeline computes full-dataset features before walk-forward slicing, which can be acceptable for strictly trailing indicators but is dangerous for market-structure event features unless every feature is causal and delayed. [VERIFIED: ai_engine/training/pipeline.py; VERIFIED: ai_engine/features/support_resistance.py] Phase 12.3 should compute specialist features per walk-forward window/prefix or add explicit one-bar delays and unit tests proving no feature reads future rows. [ASSUMED]

**Primary recommendation:** Build a separate `market_structure_liquidity` specialist using existing LightGBM/joblib/scaler patterns, evaluate it as a governed overlay against the unchanged core baseline, and only enable runtime influence if it passes a measurable uplift gate. ### Anti-Patterns to Avoid

- **Adding specialist columns to the core model without a comparison gate:** This changes the core training surface and hides whether the specialist adds value. [VERIFIED: .planning/ROADMAP.md; ASSUMED]
- **Loading specialist artifacts from the core `production.json`:** That couples rollback of core and specialist models. [VERIFIED: ai_engine/training/model_versioning.py; ASSUMED]
- **Using centered rolling features without confirmation shift:** Centered windows read future rows unless delayed; existing S/R avoids this by shifting the pivot by `swing_window`. [VERIFIED: ai_engine/features/support_resistance.py]
- **Allowing specialist BUY/SELL when core is HOLD:** This violates the user constraint that the specialist must not trade alone. [VERIFIED: user prompt]
- **Evaluating PF uplift from averaged per-window PF:** The project already records that aggregate PF must be computed from total gross profit divided by total gross loss. [VERIFIED: .planning/STATE.md; VERIFIED: ai_engine/training/walk_forward.py]

## Training/Evaluation Plan

### Training Flow

1. Build specialist features from OHLCV with explicit causal rules and `FEATURE_NAMES`. [VERIFIED: existing feature-group pattern; ASSUMED: new block]
2. Generate labels with the existing triple-barrier `LabelGenerator`, including spread/slippage and dynamic ATR if the trainer uses it. [VERIFIED: ai_engine/training/label_generator.py; VERIFIED: ai_engine/training/trainer.py]
3. Calculate expanding anchored walk-forward windows using the existing window formula and a purge gap at least equal to the label horizon. [VERIFIED: ai_engine/training/walk_forward.py]
4. For each window, fit the specialist scaler only on train rows, tune thresholds on validation rows, and evaluate on test rows. [VERIFIED: ai_engine/training/walk_forward.py; VERIFIED: ai_engine/training/trade_filter.py]
5. For each test window, compute three reports: `core_baseline`, `specialist_only`, and `core_plus_specialist_governed`. [ASSUMED]
6. Save the final-window specialist model/scaler only if the aggregate rollout gate passes. [VERIFIED: ai_engine/training/pipeline.py; ASSUMED: specialist gate]

### Leakage Controls

- Specialist features must be computed from rows available at or before the decision timestamp. [CITED: https://scikit-learn.org/stable/modules/generated/sklearn.model_selection.TimeSeriesSplit.html; ASSUMED]
- Any feature using a confirmed swing must include a confirmation delay equal to the swing window. [VERIFIED: ai_engine/features/support_resistance.py]
- Any feature using a current three-candle FVG must be shifted one row unless runtime data is proven to contain only closed candles. [ASSUMED]
- The purge gap must be at least `max(label_generator.max_candles, max_specialist_lookahead_or_confirmation_delay)`. [VERIFIED: ai_engine/training/pipeline.py; ASSUMED]
- The specialist scaler must be fresh per window and fit on train rows only. [VERIFIED: ai_engine/training/walk_forward.py]
- SHAP/feature importance should be computed on test data for generalization analysis, matching the project Phase 3 decision. [VERIFIED: .planning/STATE.md; VERIFIED: ai_engine/training/walk_forward.py]

## Rollout Gate

Enable specialist influence only if all measurable gates pass on walk-forward test windows. [ASSUMED]

| Gate | Threshold |
|------|-----------|
| Aggregate PF uplift | `Core+Specialist PF >= Core PF * 1.05`. [ASSUMED] |
| Drawdown non-regression | `Core+Specialist max_drawdown_pips <= Core max_drawdown_pips * 1.05`. [ASSUMED] |
| Trade count retention | `Core+Specialist n_trades >= Core n_trades * 0.80`, unless PF uplift is at least 15%. [ASSUMED] |

## Common Pitfalls

### Pitfall 1: Silent Feature Leakage

**What goes wrong:** Market-structure features accidentally know future swings or future FVG fills. [ASSUMED]

**Why it happens:** Swing/FVG logic often uses centered windows, forward fill, or fill ratios that are easy to compute from the full DataFrame. [VERIFIED: ai_engine/features/support_resistance.py; ASSUMED]

**How to avoid:** Compute with previous confirmed levels only, shift event flags when necessary, and test that changing future rows does not change past features. [VERIFIED: ai_engine/features/support_resistance.py; ASSUMED]

**Warning signs:** Unrealistically high PF, nearly perfect FVG/sweep precision, or per-window performance collapsing when features are recomputed per window. [ASSUMED]

### Pitfall 2: Specialist Becomes a Hidden Replacement

**What goes wrong:** Core+Specialist code changes the baseline decision path even when the specialist is disabled. [ASSUMED]

**Why it happens:** The specialist is folded into the same global score calculation instead of applied as a post-core governed overlay. [VERIFIED: ai_engine/prediction/ensemble.py; ASSUMED]

**How to avoid:** Add disabled/unavailable tests asserting core-only signal equality. [ASSUMED]

**Warning signs:** Baseline `action`, `confidence`, or `final_aggregation` changes with `specialist_enabled=False`. [ASSUMED]

### Pitfall 3: Improving PF by Removing Too Many Trades

**What goes wrong:** A veto-heavy specialist increases PF but makes the bot too inactive for demo validation. [ASSUMED]

**Why it happens:** PF ignores opportunity cost and can improve when low-confidence trades are over-filtered. [ASSUMED]

**How to avoid:** Include trade-count retention and per-window robustness in the rollout gate. [ASSUMED]

**Warning signs:** `n_trades` falls below 80% of baseline while PF improvement is modest. [ASSUMED]

### Pitfall 4: Specialist and Core Share the Same Scaler

**What goes wrong:** Specialist features are transformed with a core scaler trained on different columns. [ASSUMED]

**Why it happens:** Existing runtime has one `FeatureScaler` for core model features. [VERIFIED: ai_engine/prediction/ensemble.py]

**How to avoid:** Save and load `specialist_scaler.pkl` with the specialist feature list. [ASSUMED]

**Warning signs:** Missing feature columns, unexpected zero fills, or shape mismatch during specialist inference. [VERIFIED: ai_engine/prediction/ensemble.py; ASSUMED]

### Pitfall 5: Production Pointer Collision

**What goes wrong:** Promoting a specialist overwrites the core model pointer or copies files into the core saved-model root. [ASSUMED]

**Why it happens:** Existing `update_production_pointer()` copies core model files into the base directory for backward compatibility. [VERIFIED: ai_engine/training/model_versioning.py]

**How to avoid:** Use a specialist-specific pointer function that does not copy to the core base directory. [ASSUMED]

**Warning signs:** `xgboost_gold.pkl`, `lightgbm_gold.pkl`, or `feature_scaler.pkl` timestamps change during specialist promotion. [ASSUMED]
