# Phase 12.3 Plan 03 Summary: Runtime Specialist Overlay

Integrated the specialist as an optional runtime overlay without breaking the core ensemble baseline.

One-liner: `SpecialistRuntime` now loads the isolated specialist bundle as a no-op-safe adapter, `EnsemblePredictor` logs specialist fields only when artifacts are available, and the final governance step can confirm, weaken, veto, or reject specialist-only trade attempts.

## What Changed

- Added `ai_engine/prediction/specialist.py` with `SpecialistPrediction` and `SpecialistRuntime` for isolated specialist loading and no-op fallback behavior when artifacts are absent.
- Extended `ensemble.py` to request opt-in specialist features, lift the primary specialist signal into the final governance call, and expose `core_action`, `core_confidence`, `specialist_score`, `specialist_confidence`, `specialist_reason`, and `specialist_effect` in the final aggregation payload.
- Kept baseline runtime behavior exact when the specialist is disabled or unavailable by leaving the public signal shape unchanged in that path.
- Reused the Phase 12.1 `DecisionGovernor` specialist hooks so the overlay can confirm or weaken a core signal, veto strong conflicts, and never create a standalone trade from a core `HOLD`.
- Added focused tests for missing-artifact adapter fallback, baseline equality, specialist logging, and no-trade-alone enforcement.
