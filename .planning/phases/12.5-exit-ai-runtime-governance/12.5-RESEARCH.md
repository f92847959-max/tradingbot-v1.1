# Phase 12.5: Exit AI Runtime Integration & Governance - Research

**Researched:** 2026-04-24
**Domain:** Governed trade-management runtime / risk-reducing overlays / audit and reconciliation
**Confidence:** HIGH

---

## Summary

Phase 12.5 is the live-runtime follow-up to 12.4. The repo already has the required
broker-side and lifecycle orchestration layers:

- `order_management/order_manager.py` owns `modify_position`, `close_trade`, and periodic position checks
- `order_management/position_monitor.py` owns open/closed transition detection
- `exit_engine/trailing_manager.py` and `partial_close.py` already define the monotonic risk-reduction semantics

That means the correct integration pattern is a governed overlay, not a replacement.
Exit AI should advise or trigger only risk-reducing actions through existing execution
paths. It must never widen a stop, create a trade, or bypass the broker/DB reconciliation
logic already centralized in `OrderManager`.

**Primary recommendation:** Create a runtime Exit-AI advisor that loads 12.4 artifacts,
evaluates open positions during the existing management loop, and outputs one of:

- `NOOP`
- `TIGHTEN_SL`
- `PARTIAL_CLOSE`
- `FULL_EXIT`

The advisor is allowed to influence the existing trade lifecycle only when:

- artifacts are present,
- confidence/governance thresholds pass,
- and the proposed action reduces or finalizes risk.

Everything else must degrade to no-op.

## Architecture Patterns

### Pattern 1: Advisor, not autonomous execution engine

Add a thin runtime advisor that proposes a bounded action. Let `OrderManager` remain the
single place where broker-side effects happen.

### Pattern 2: Hard no-risk-widening guardrail

Any proposed action must pass these gates:

- new stop loss is only accepted if it is more favorable than the current stop
- partial close fires only through the existing partial-close semantics
- full exit routes through the existing close path
- missing artifacts, low confidence, or invalid proposal -> no-op

### Pattern 3: Audit first

Every Exit-AI recommendation needs structured logging:

- baseline context
- recommended action
- confidence / reason
- whether the action was applied, rejected, or skipped
- final broker outcome

### Pattern 4: Drift and reconciliation

Runtime logging should make it possible to answer:

- what Exit-AI recommended
- what baseline exit logic would have done
- what was actually executed
- how the trade eventually resolved

This enables retraining and operator review.

## Common Pitfalls

### Pitfall 1: Stop-loss widening by arithmetic bug

If a calculated stop is applied without direction-aware monotonicity checks, BUY stops can
move down or SELL stops can move up.

**Avoidance:** compare against current stop and reject any non-favorable update before
calling `modify_position`.

### Pitfall 2: Repeated partial closes

If Exit AI can repeatedly emit partial close recommendations without shared tracking, one
position may be closed in multiple unintended chunks.

**Avoidance:** reuse or extend the existing partial-close state-tracking semantics.

### Pitfall 3: Broker/DB drift after runtime failure

Any new runtime branch that modifies or closes positions can amplify state divergence if it
bypasses `OrderManager` persistence/reconciliation logic.

**Avoidance:** route all side effects through `OrderManager` and reuse reconciliation queues.

### Pitfall 4: Advisor becomes hidden entry logic

If runtime outputs are allowed to create new trades or reverse positions, Exit AI stops
being an exit specialist and becomes an unmanaged strategy layer.

**Avoidance:** hard action whitelist plus tests asserting no entry creation.
