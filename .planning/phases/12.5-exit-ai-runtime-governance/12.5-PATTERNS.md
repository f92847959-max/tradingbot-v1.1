## Reuse Patterns

### 1. Optional specialist runtime

Use optional specialist behavior as the pattern, not mandatory hard-fail runtime.

Implication for 12.5:
- artifact load failure must result in no-op, not trade-management failure
- recommendation objects should stay structured and explicit

### 2. Single side-effect owner

Use `OrderManager` as the only side-effect owner.

Implication for 12.5:
- advisor returns bounded proposals only
- `modify_position`, `close_trade`, and tracked partial-close paths remain centralized

### 3. Monotonic risk reduction

Use `SmartTrailingManager` and `PartialCloseManager` semantics as the risk guardrail pattern.

Implication for 12.5:
- `TIGHTEN_SL` must obey monotonic direction-aware improvement
- `PARTIAL_CLOSE` must stay one-shot and tracked

### 4. Reconciliation-aware logging

Use the existing broker/DB divergence and callback handling in `OrderManager` / `PositionMonitor`.

Implication for 12.5:
- audit data should remain available even when modify/close operations fail and retry
- new runtime behavior must not short-circuit reconciliation queues

## Avoid

- Do not let the advisor call broker APIs directly.
- Do not allow runtime action proposals outside the bounded manifest.
- Do not treat missing artifacts as fatal.
- Do not lose recommendation context when broker/DB retry paths are triggered.
