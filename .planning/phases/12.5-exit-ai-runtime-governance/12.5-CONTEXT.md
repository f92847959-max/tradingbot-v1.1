# Phase 12.5: Exit AI Runtime Integration & Governance - Context

**Gathered:** 2026-04-24
**Status:** Ready for planning
**Mode:** Smart-discuss defaults applied for autonomous planning

<domain>
## Phase Boundary

Phase 12.5 wires the trained Exit-AI specialist from 12.4 into the live trade-management
loop as a governed overlay.

The scope is:
- runtime artifact loading and advisor contract
- integration into the existing trade-management loop
- risk-reducing application of `TIGHTEN_SL`, `PARTIAL_CLOSE`, `FULL_EXIT`
- structured audit logging and drift/reconciliation support

Out of scope:
- any new entry generation
- stop widening or risk increase
- bypassing existing broker/DB orchestration in `OrderManager`

</domain>

<decisions>
## Implementation Decisions

### Runtime Role
- Exit AI is an advisor attached to the existing order-management path.
- `OrderManager` remains the only component allowed to create broker side effects.
- Missing artifacts, disabled config, invalid action proposals, or low confidence all degrade to no-op.

### Action Guardrails
- `TIGHTEN_SL` may only move the stop in the favorable direction.
- `PARTIAL_CLOSE` must reuse tracked one-shot partial-close semantics.
- `FULL_EXIT` must reuse the existing `close_trade` path.
- Entry actions, reverse-position actions, and stop widening are forbidden.

### Logging and Audit
- Each recommendation must record baseline context, action, confidence, and whether it was applied.
- Broker-side result and final trade outcome must be linkable back to the recommendation.
- Logs must be good enough for retraining and drift review.

### Reconciliation
- New runtime branches must respect the existing orphan/reconciliation safeguards in `OrderManager`.
- Exit-AI recommendations must never bypass the persistence and retry logic already built around modify/close flows.

### the agent's Discretion
- Exact module location of the runtime advisor
- Whether audit data extends existing order logs or introduces a narrowly-scoped new table/field set
- Exact confidence thresholds, as long as guardrails and no-op fallback remain explicit

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `order_management/order_manager.py` owns `modify_position`, `close_trade`, and periodic `check_positions`
- `order_management/position_monitor.py` owns close detection and reconciliation-sensitive lifecycle tracking
- `exit_engine/trailing_manager.py` already defines monotonic stop-tightening semantics
- `exit_engine/partial_close.py` already defines one-shot TP1 close semantics
- 12.1 governance patterns already define how calibrated gates and audit reasons should be represented

### Established Patterns
- optional components degrade to no-op when unavailable
- broker is the source of truth, but DB audit and reconciliation matter
- focused pytest files per behavior area are preferred over one giant integration file

</code_context>

<specifics>
## Specific Ideas

- Model output should likely be wrapped in a small advisor result object with fields like `action`, `confidence`, `proposed_stop`, `close_fraction`, `reason`.
- `OrderManager.check_positions()` is the cleanest initial hook because it already runs periodic exit-management logic.
- Drift monitoring should compare three things: baseline recommendation, Exit-AI recommendation, and applied broker action.

</specifics>
