# Phase 12.6: Existing AI Autonomy Distillation - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning
**Mode:** Research-backed autonomous planning

<domain>
## Phase Boundary

Phase 12.6 improves the existing AI stack by distilling decisions from the current deterministic teacher chain into a learned decision head.

The scope is:
- causal teacher snapshots per decision tick
- hierarchical label schema for decision intent and guard blocks
- offline-first distillation dataset for the current AI stack
- calibrated decision-head training and walk-forward comparison
- shadow-mode and guarded runtime rollout inside the existing signal path

Out of scope for 12.6:
- new broker execution semantics
- broader action space than `SELL/HOLD/BUY`
- removing `DecisionGovernor`, `StrategyManager`, calendar guards, or `RiskManager`
- replacing Exit AI, Specialist AI, or MiroFish with a separate service

</domain>

<decisions>
## Implementation Decisions

### Autonomy Role
- The learned model upgrades the existing entry-decision path; it is not a new standalone system.
- Existing deterministic systems remain teachers during training and hard runtime guards after rollout.
- The runtime chain must remain `kill switch / event guard -> AI signal -> strategy filter -> risk approval -> broker execution`.

### Labeling Strategy
- `preliminary_action`, `policy_action`, `final_action`, `block_stage`, and `block_codes` remain separate fields.
- Blocked actions must not be flattened into generic `HOLD`.
- Soft teacher probabilities from the current ensemble should be preserved for distillation and calibration analysis.

### Model and Artifact Shape
- The decision head stays `SELL/HOLD/BUY` only in this phase.
- Artifacts live under an isolated autonomy-head root with version metadata and `production.json`.
- Calibration and threshold formats should reuse the Phase 12.1 contract.

### Rollout Discipline
- Rollout stages are `shadow`, `agreement_guarded`, and `primary_with_challenger`.
- Champion and candidate must score the same snapshot.
- Only one selected raw signal may enter the existing downstream execution path.

### the agent's Discretion
- exact feature names for teacher snapshot exports
- exact artifact root naming under the existing saved-model structure
- precise promotion thresholds, as long as they remain baseline-relative and calibrated

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ai_engine/prediction/ensemble.py` already computes decision-state features, multi-timeframe aggregation, and governance handoff
- `ai_engine/governance/decision_governor.py` already defines the final runtime gate semantics
- `trading/trading_loop.py` contains the correct decision-time capture seam and the hard external guard chain
- `strategy/strategy_manager.py` is already a deterministic teacher stage for trade-score and session/regime gating
- `risk/risk_manager.py` returns structured approval and failed checks that should become teacher labels
- `database/models.py` + `database/repositories/governance_repo.py` already persist structured governance events
- `monitoring/model_monitor.py` and `ai_engine/governance/promotion.py` provide promotion and degradation patterns

### Established Patterns
- optional learned runtime components default to safe fallback when artifacts are missing
- walk-forward validation and train-only fitting are mandatory for performance claims
- calibration and threshold artifacts are JSON-friendly and versioned
- isolated artifact roots are preferred for optional overlays and challengers

</code_context>

<specifics>
## Specific Ideas

- Add an append-only `decision_snapshots` capture layer rather than reconstructing the dataset only from `signals` or trades.
- Extend `GovernanceDecision.evaluation_summary` for rollout metadata instead of creating a second audit table.
- Use a light `LightGBM` multiclass head plus a simpler challenger baseline for the same snapshot schema.
- Carry `failed_checks` and structured `block_codes` through into rollout analytics.

</specifics>

<deferred>
## Deferred Ideas

- online self-learning or DAgger-style active relabeling
- expanded action spaces beyond `SELL/HOLD/BUY`
- broker-side autonomous execution policies
- replacing downstream guard systems with AI-only safety logic

</deferred>
